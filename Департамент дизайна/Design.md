# Design.md — Правила Департамента Дизайна

## Миссия

Создавать красивые, функциональные дизайны для пяти продуктовых направлений и деплоить их на Vercel как продакшн-готовые веб-страницы.

## Продуктовые направления

1. **Презентации** — слайдовые веб-презентации с анимациями и переходами
2. **Сметы** — структурированные таблицы расчётов с интерактивными блоками
3. **ТЗ (Технические задания)** — документы с навигацией, секциями, прогрессом чтения
4. **КП (Коммерческие предложения)** — продающие лендинги с офферами и CTA
5. **Рекламные креативы** — статичные/анимированные баннеры, лендинги под трафик

---

## Технологический стек

**Обязательно:**
- **Next.js 15+** (App Router) — фреймворк
- **TypeScript** — строгая типизация
- **Tailwind CSS** — стилизация
- **shadcn/ui** — базовые UI-компоненты
- **Framer Motion** — анимации и переходы
- **Lucide React** — иконки

**По задаче:**
- **MDX** — для контента документов (ТЗ, КП)
- **Recharts / Tremor** — для графиков в сметах
- **next/font** — типографика (Inter, Manrope, или премиум-шрифты)
- **next/image** — оптимизация изображений

**Деплой:**
- **Vercel** — единственная целевая платформа
- Каждый проект деплоится отдельно или как маршрут в общем монорепо (выбор по контексту)

## AI-инструменты для дизайна

### Google Stitch — основной генератор UI и веба
- URL: https://stitch.withgoogle.com
- Назначение: быстрая AI-генерация интерфейсов из текстового брифа и/или эскиза. Используется на этапе **концепции и каркаса**, до верстки на Next.js.
- Когда применять:
  - Лендинги под КП и рекламные креативы (быстрый прототип за минуты)
  - Веб-страницы презентаций и ТЗ (первичная композиция секций)
  - Мобильные UI и дашборды для смет
  - Любая задача, где нужно за 10–15 минут получить 2–3 визуальных варианта для согласования
- Workflow со Stitch:
  1. Сформулировать промпт: продукт, ЦА, тон (бизнес / премиум / молодёжный), ключевой экран, обязательные секции, CTA.
  2. Сгенерировать 2–3 варианта в Stitch.
  3. Согласовать направление с пользователем через оркестратора.
  4. Экспортировать выбранный вариант в **Figma** (для доводки) или взять HTML/CSS как референс.
  5. Реализовать на Next.js + Tailwind + shadcn/ui по правилам стека ниже.
- Что НЕ делать:
  - Не отдавать Stitch-экспорт как финал — он только концепт, продакшн всегда на Next.js.
  - Не пропускать согласование концепции — экономит часы переделок.

### Вспомогательные
- **Figma** — доводка макета после Stitch, работа с дизайн-системой
- **Miro** (MCP) — оргсхемы, mind map, мудборды, ревью макетов
- **v0.dev** — альтернативный генератор React/Tailwind кода, когда нужен сразу компонент

## Принципы дизайна

### 1. Визуальная иерархия превыше всего
Заголовок должен бить, подзаголовок — пояснять, тело — продавать или раскрывать. Размеры, веса, цвета — всё работает на иерархию.

### 2. Пространство — это часть дизайна
Не бояться воздуха. `py-24`, `py-32` для секций. Никакой плотности ради плотности.

### 3. Типографика — фундамент
- Заголовки: крупно, жирно, с tracking-tight
- Тело: 16–18px, leading-relaxed (1.6–1.75)
- Минимум 2 шрифта на проект, максимум 3
- Никогда не использовать дефолтный Times/Arial

### 4. Цветовая дисциплина
- 1 основной цвет бренда
- 1 акцент (контрастный)
- Нейтральная шкала (zinc / slate / stone)
- Без радуги. Без больше 5 цветов на проекте.

### 5. Анимации с целью
Анимация должна нести смысл: фокус, переход, обратная связь. Никакого "анимировать ради анимации". Длительность: 200–600ms. Easing: `ease-out` для входа, `ease-in` для выхода.

### 6. Респонсив — обязателен
Mobile-first. Брейкпоинты Tailwind: `sm`, `md`, `lg`, `xl`. Проверять на 375px (iPhone SE), 768px (iPad), 1440px (десктоп).

### 7. Производительность
- Lighthouse Performance ≥ 90
- Картинки только через `next/image`
- Шрифты через `next/font` (без CLS)
- LCP < 2.5s

## Правила по направлениям

### Презентации
- Полноэкранные слайды (`h-screen`)
- Навигация: клавиши (←/→), скролл, кнопки
- Переходы между слайдами через Framer Motion (`AnimatePresence`)
- Прогресс-бар вверху или счётчик слайдов
- Тёмные/светлые темы — обе доступны
- Поддержка экспорта в PDF (через `print` стили)

### Сметы
- Сетка-таблица: позиции, кол-во, цена, сумма
- Итоги: фиксированный футер или sticky
- Категории сворачиваются/разворачиваются
- Подсветка ключевых сумм (НДС, итого, скидка)
- Возможность переключать валюту/единицы
- Минимум декора, максимум читаемости

### ТЗ
- Левый sidebar с навигацией по разделам
- Прогресс чтения (scroll progress)
- Якорные ссылки на каждый заголовок
- Код-блоки с подсветкой синтаксиса
- Версионирование документа (v1.0, дата)
- Print-friendly стили

### КП
- Hero с офером и CTA в первом экране
- Социальные доказательства (кейсы, отзывы, логотипы)
- Структура: проблема → решение → продукт → цена → CTA
- Сравнительные таблицы тарифов
- FAQ-блок
- Финальный CTA с формой или контактами

### Рекламные креативы
- Лендинг под один оффер (one-page, без меню)
- Скорость загрузки — приоритет №1
- Один CTA, повторяющийся 3–5 раз по странице
- A/B варианты через query-параметры или поддомены
- Pixel-perfect под мобильный (большинство трафика — мобайл)

## Workflow на каждую задачу

1. **Бриф.** Получить вводные: продукт, тематика, контент, ЦА, дедлайн, референсы
2. **Концепция через Stitch.** Сгенерировать 2–3 варианта UI в Google Stitch по брифу, выбрать направление, согласовать с пользователем
3. **Структура.** Создать `app/` + базовые компоненты, поставить зависимости
4. **Контент-каркас.** Заполнить страницы реальным контентом из Stitch-концепта (тексты — от Копирайтера через оркестратора)
5. **Стилизация.** Применить дизайн-систему: цвета, типографика, отступы
6. **Анимации.** Добавить переходы и микроинтеракции
7. **Респонсив.** Проверить и поправить на мобильном и планшете
8. **Оптимизация.** Lighthouse-чек, оптимизация картинок и шрифтов
9. **Деплой.** `vercel deploy --prod`, отдать ссылку

## Стандарты кода

- **TypeScript strict mode**: всегда `strict: true`
- **Без `any`**. Если нужен — `unknown` + type guard
- **Компоненты:** функциональные, server-first (`'use client'` только при необходимости)
- **Именование:** компоненты PascalCase, файлы kebab-case или PascalCase для компонентов
- **CSS:** только Tailwind, без inline-стилей и CSS-модулей (исключение — глобальные стили в `globals.css`)
- **Без комментариев в коде**, если код самодокументируем. Только WHY-комментарии, когда логика неочевидна

## Деплой на Vercel

1. Проект подключается к Vercel через CLI или GitHub
2. Команды:
   - `vercel` — preview-деплой
   - `vercel --prod` — продакшн
3. Переменные окружения — через Vercel Dashboard
4. Кастомный домен — через настройки проекта
5. Каждый продакшн-деплой сохраняется в истории

## Чего НЕ делать

- ❌ Не использовать `<img>` — только `next/image`
- ❌ Не подключать тяжёлые библиотеки ради одной фичи (Material UI, Bootstrap)
- ❌ Не делать дизайн "как у всех" — каждый проект имеет характер
- ❌ Не оставлять `console.log` и закомментированный код
- ❌ Не деплоить с ошибками типов или lint
- ❌ Не использовать emoji в дизайнах, если этого явно не просили
- ❌ Не плодить лишние абстракции — три похожие строки лучше преждевременной абстракции
