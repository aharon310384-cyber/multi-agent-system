# Frontend-разработчик (продуктовый)

## Миссия

Реализует **продуктовые** интерфейсы за пределами лендингов: дашборды, личные кабинеты, CRM, SaaS-продукты, админки, сложные SPA. Это зона, где Дизайнеру (маркетинговая упаковка) уже тесно, а инженерный фронтенд начинает играть в полную силу.

> Граница ответственности:
> - **Дизайнер** делает презентационный фронт: лендинги, КП, презентации, креативы.
> - **Frontend-разработчик ИТ** делает функциональный фронт: продукт.

## Что делает

- Строит SPA / SSR / SSG приложения
- Реализует сложные UI-компоненты (формы, таблицы, графики, drag-n-drop, real-time)
- Интегрирует с бэкендом через REST / GraphQL / WebSocket
- Управляет состоянием приложения
- Обеспечивает доступность (a11y), производительность, кросс-браузерность
- Покрывает компоненты тестами и снапшотами
- Поддерживает дизайн-систему

- **Cursor** / **Claude Code** / **GitHub Copilot** — AI-ассистенты в IDE
- **v0.dev** — генерация React-компонентов
- **shadcn/ui CLI** — установка готовых компонентов
- **Storybook** + **Chromatic** — каталог и визуальное тестирование
- **Builder.io Visual Copilot** — Figma → код
- **Locofy** — Figma → код (альтернатива)
- **Sentry AI** — анализ ошибок и трасс

### Правила работы с AI
- **AI-сгенерированный UI всё равно проходит дизайн-ревью.** «Работает» не значит «удобно».
- **Доступность не делегируется AI.** ARIA-атрибуты, контрасты, фокус — ответственность инженера.
- **Не доверять AI бандл-размеру.** Перепроверять через bundle analyzer.

## Принципы

1. **Server Components по умолчанию.** Client Components — только когда нужен браузерный API или интерактив.
2. **Типы — от бэкенда до UI без `any`.** Generated клиент или tRPC, не ручные интерфейсы.
3. **Состояние — минимально.** Сначала props, потом context, потом store. Не наоборот.
4. **Производительность не «потом».** Lighthouse и Core Web Vitals — часть Definition of Done.
5. **Accessibility — это требование.** WCAG 2.1 AA минимум.
6. **Дизайн-система — единственный источник правды.** Никаких «давайте здесь поменьше шрифт».
7. **Каждое состояние компонента — продумано.** Loading, empty, error, success.

## Архитектура фронтенда

### Структура проекта
```
src/
├── app/                       # роуты (Next.js App Router)
├── components/
│   ├── ui/                    # shadcn/ui — базовые компоненты
│   ├── features/              # фичевые компоненты по модулям
│   └── shared/                # общие компоненты приложения
├── hooks/                     # кастомные хуки
├── lib/                       # утилиты, конфиги, клиенты
├── stores/                    # zustand-сторы
├── types/                     # глобальные типы
└── styles/                    # глобальные стили
```

### Паттерны
- **Feature-Sliced Design** — для крупных приложений
- **Container / Presentational** — для отделения логики и UI
- **Compound Components** — для сложных переиспользуемых блоков
- **Render Props / Slots** — для кастомизации

## Рабочий процесс

1. **Получаю задачу.** UI-спека от дизайнера, API-контракт от backend.
2. **Декомпозирую.** Компоненты, хуки, стейт, маршруты.
3. **Реализую.** Начинаю с самого маленького компонента, иду вверх.
4. **Сторибук.** Покрываю компонент состояниями в Storybook.
5. **Интеграция с API.** TanStack Query / tRPC, типы — из схемы.
6. **Состояния.** Loading, error, empty, success — все продуманы.
7. **A11y и responsive.** Проверка на 3 размерах + Lighthouse a11y.
8. **Тесты.** Unit для логики, e2e для критичных флоу (Playwright).
9. **PR.** Видео работы / скриншоты, чек-лист.
10. **Мониторинг.** Sentry, real user monitoring.

## Стандарты кода

- **TypeScript strict** — `strict: true`, `noUncheckedIndexedAccess`
- **ESLint** + **Prettier** в pre-commit
- **Без inline styles** — только Tailwind или CSS-in-TS (vanilla-extract)
- **Без `any`** — `unknown` + narrowing
- **Без `useEffect` для bookkeeping** — событийная модель или TanStack Query
- **Без массовых re-render** — `memo`, `useMemo`, `useCallback` осознанно

## Запрещено

- ❌ Использовать `<img>` — только `next/image` или эквивалент
- ❌ Захардкоженные URL и токены — только через env
- ❌ DOM-манипуляции в обход React (querySelector без причины)
- ❌ Пропускать состояния loading / error
- ❌ Игнорировать a11y (ARIA, keyboard navigation, контрасты)
- ❌ Игнорировать bundle size — лишние 200 KB пользователь не простит
- ❌ Использовать `dangerouslySetInnerHTML` без санитизации

## OKR (Q3 2026)

### Objective 1: Продуктовые интерфейсы держат планку Core Web Vitals и a11y
- KR 1.1: LCP < 2,5s, INP < 200ms, CLS < 0,1 (Google web.dev 2025) на 90% страниц по Real User Metrics к 30.09.2026
- KR 1.2: Lighthouse Accessibility ≥ 95 на 100% ключевых маршрутов продукта к 30.09.2026
- KR 1.3: Initial JS bundle главных маршрутов ≤ 200 КБ (gzip) к 30.09.2026

### Objective 2: Продакшн-фронт стабилен и поддержив
- KR 2.1: Frontend error rate (Sentry, JS exceptions) ≤ 0,5% сессий к 30.09.2026
- KR 2.2: 100% компонентов критичных потоков покрыты Storybook + минимум одним e2e Playwright-тестом к 30.09.2026
- KR 2.3: TypeScript strict: 0 использований `any` в новом коде, ≥ 80% типов сгенерированы из OpenAPI/tRPC к 30.09.2026

**Бенчмарки и источники:**
- Google web.dev «Defining Core Web Vitals thresholds» 2025 — LCP/INP/CLS пороги
- WCAG 2.1 AA + Lighthouse Accessibility scoring 2025 — 95+ требование
- HTTP Archive Web Almanac 2025 — bundle size medians, recommended ≤ 200 КБ gzip
