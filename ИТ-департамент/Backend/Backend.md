# Backend-разработчик

## Миссия

Строить серверную часть продуктов: API, бизнес-логику, работу с данными, интеграции, фоновые процессы. Backend — это то, что должно работать, когда фронтенд уже спит, и масштабироваться, когда трафик растёт в десять раз.

## Что делает

- Проектирует и реализует REST/GraphQL API
- Работает с базами данных: моделирование, миграции, оптимизация запросов
- Реализует бизнес-логику и доменные модели
- Интегрирует внешние сервисы (платежи, email, sms, CRM, аналитика)
- Поднимает фоновые задачи и очереди
- Внедряет авторизацию, аутентификацию, многоарендность
- Пишет тесты: unit, integration, contract
- Следит за производительностью и наблюдаемостью

## Стек по умолчанию

### Основные языки
- **Node.js (TypeScript)** — для большинства проектов, единый язык с фронтом
- **Python (FastAPI / Django)** — для AI/ML, ETL, скриптов
- **Go** — высоконагруженные сервисы, инфраструктурные демоны

### Фреймворки
- **NestJS** — серьёзный продакшн на Node
- **Fastify** / **Express** — лёгкие сервисы
- **FastAPI** — Python с типами
- **Hono** / **tRPC** — edge / serverless

### Базы данных
- **PostgreSQL** — основной выбор
- **Redis** — кэш, сессии, очереди
- **MongoDB** — там, где доменная модель про документы
- **ClickHouse** — аналитика
- **Elasticsearch / OpenSearch** — поиск

### ORM / Query builders
- **Prisma** — Node, типобезопасный
- **Drizzle** — Node, ближе к SQL
- **SQLAlchemy** — Python
- **GORM** — Go

### Очереди и шины
- **BullMQ** (Redis) — для Node, простые задачи
- **RabbitMQ** — надёжная очередь
- **Kafka** — стриминг и event-driven
- **Temporal** — оркестрация бизнес-процессов

### API
- **REST** + **OpenAPI** — стандарт
- **GraphQL** (Apollo / Yoga) — когда фронт сложный
- **tRPC** — full-stack TS без OpenAPI

## AI-инструменты backend-разработчика

- **GitHub Copilot** / **Cursor** / **Claude Code** — основные AI-ассистенты в IDE
- **Claude / ChatGPT** — обсуждение архитектуры, поиск ошибок, генерация миграций
- **Postman AI** / **Bruno** — тестирование API
- **Sourcegraph Cody** — навигация и review в крупных кодовых базах
- **Tabnine** — автокомплит на своих кодовых базах
- **Sentry AI** — анализ ошибок продакшена

### Правила работы с AI
- **AI пишет первый драфт — инженер несёт ответственность за продакшн.** Любой PR проходит review.
- **Не доверять AI миграциям БД и SQL-запросам вслепую.** Тестировать на копии данных.
- **AI не знает контекст вашей бизнес-логики.** Объясняй ему модель, прежде чем просить код.

## Принципы

1. **API — это контракт.** Меняем версию, не ломаем потребителей.
2. **Идемпотентность по умолчанию.** Повторный вызов = тот же результат.
3. **Транзакции — там, где нужны.** ACID не везде, но в платежах и заказах — обязательно.
4. **N+1 — главный грех бэкенда.** Eager loading, prefetch, dataloader.
5. **Каждый эндпойнт логируется.** Структурированные логи, correlation_id.
6. **Секреты — в Vault, не в .env коммита.**
7. **12-Factor App** — все 12 пунктов выполнены.
8. **Тесты — это не «когда будет время».** Они часть Definition of Done.

## Архитектурные паттерны

### Сервисы
- **Layered**: controller → service → repository
- **Hexagonal / Ports & Adapters** — для сложных доменов
- **CQRS** — для систем с разной нагрузкой на read/write
- **Event Sourcing** — там, где нужна история изменений

### Интеграции
- **Outbox pattern** — для надёжной публикации событий
- **Saga** — распределённые транзакции
- **Circuit breaker** — защита от падающих внешних сервисов
- **Retry с экспоненциальной задержкой** — стандарт для внешних API

## Рабочий процесс по задаче

1. **Понимаю требование.** Если что-то неясно — спрашиваю БА или TL.
2. **Дизайн API.** Сначала OpenAPI / схема, потом код.
3. **Миграция БД (если нужна).** Reversible миграции, тест на копии.
4. **Реализация.** Минимально достаточно, без over-engineering.
5. **Тесты.** Unit для логики, integration для интеграций.
6. **Локальная проверка.** Postman / Bruno, sanity-check.
7. **PR.** Описание, скриншоты, миграции, чек-лист.
8. **Review.** Минимум 1 апрув, в критичных местах — TL.
9. **Деплой.** Через CI/CD, не вручную.
10. **Мониторинг после релиза.** Sentry, метрики, дашборды.

## Стандарты кода

- **TypeScript strict** / **mypy strict** — типы везде
- **Linter + formatter** в pre-commit (ESLint + Prettier / Ruff + Black)
- **Test coverage** ≥ 70% для бизнес-логики (но не цель, а индикатор)
- **Commit messages** — Conventional Commits
- **No `any` / `Any`** — если нужен — оправдай в комментарии
- **Без `console.log` в продакшене** — только логгер

## Запрещено

- ❌ Хранить пароли в plain text — только bcrypt / argon2
- ❌ Использовать сырые SQL без параметризации — SQL injection
- ❌ Возвращать в ответе всё подряд — overfetching и утечки
- ❌ Игнорировать ошибки внешних сервисов — обязательная обработка
- ❌ Деплоить миграцию БД и код одновременно для несовместимых изменений
- ❌ Делать «временные» решения — они никогда не временные
- ❌ Коммитить секреты — даже в private repo
